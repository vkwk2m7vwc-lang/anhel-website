### 4. Валидация квиза — email, телефон, consent

  - Последний шаг: `data.consent === true`

### 5. Честный submit — убран ложный success

**Статус:** done
**Что было:** `setSubmitted(true)` без единого fetch. Экран «Заявка принята» для всех, кто кликнул — обман пользователя (он ждёт звонка, никто не получает заявку).
**Что стало — содержательное решение:** вариант (а) из пакета правок, усиленный mailto-отправкой.

**Как работает:**

1. Над формой висит persistent notice banner: «Форма в разработке. Нажатие «Открыть в почтовом клиенте» сформирует письмо с введёнными данными — останется нажать «Отправить» в вашем почтовом клиенте. Не работает? Свяжитесь напрямую: [phone] · [email]»
2. Финальная кнопка: «Отправить заявку» → **«Открыть в почтовом клиенте»**
3. При клике: `buildMailtoHref(data)` генерирует `mailto:info@anhelspb.com?subject=…&body=…` с предзаполненным письмом из всех непустых полей (структурированно — по секциям Контакты / Объект / Гидравлика / Насосы / Опции / Дополнительно, с RU-подписями для всех enum'ов)
4. `window.location.href = mailto` — открывается почтовый клиент пользователя с готовым письмом
5. Secondary state `QuizSuccess` полностью переписан: «Открыт почтовый клиент» → «Проверьте почтовый клиент — в нём уже письмо с вашими параметрами» + fallback на tel/mailto если клиент не открылся

**Почему так, а не просто disabled:**

- Пользователь уже вложил 2–3 минуты в заполнение — выбрасывать это в никуда жестоко. Mailto отдаёт данные ему же
- Если у пользователя нет mail client — QuizSuccess показывает телефон и email как явный fallback
- Zero backend surface — когда приедет Resend/Telegram, удаляется `mailto.ts`, заменяется `goNext()` на `fetch(...)`, остальное не трогается

**Consent-текст в шаге 6** синхронизирован с новой надписью кнопки: «Нажимая «Открыть в почтовом клиенте», я даю согласие…»

**Новые файлы:**

- `src/lib/contacts.ts` — единая точка для телефона / email / адреса (используется в Header, Footer, QuizSuccess, notice banner, MobileMenu)
- `src/components/product-page/quiz/mailto.ts` — построение письма из `QuizData`

## Open issues from user observations

Три живых наблюдения заказчика — провёл кодовый анализ; рекомендации помечены как gap для Group 2/3.

### Наблюдение 1 — Секция 3 «Как срабатывает»: текст и башня в разных реальностях, скролл слишком долгий

**Код-анализ** (`HowItWorksSection.tsx`, `LakhtaSteps.tsx`, `LakhtaScene.tsx`):

- **Длина прокрутки.** В `HowItWorksSection.tsx:110` стоит `end: "+=400%"` — это 4× высоты viewport'а для 6 шагов. В комментарии к правке написано: «Trimmed from 600% → 400% (≈1.5× faster than the initial landing)». Т.е. это уже сокращённая версия, и заказчик всё равно воспринимает как медленно. Для комфортного темпа 6 шагов нужно ~300% или даже 250% — 0.4–0.5 viewport на шаг.
- **Композиция 40/60.** `grid-cols-[minmax(320px,2fr)_minmax(0,3fr)]` — текст слева, башня справа, в одной grid-ячейке. Визуально сетка есть, но перекрёстных линий/якорей между колонками нет — глаз воспринимает их как параллельные, несвязанные блоки.
- **Высота.** `LakhtaScene` имеет `aspect-[600/1000]` (портрет, высокая). `LakhtaSteps` — вертикальный rail из 6 шагов с активной подсветкой. Оба элемента start-aligned (`self-start`) но свободно тянутся вниз по своему содержимому. При активном шаге 01 — текст короткий, а башня — полная высота; рассинхрон налицо.

**Рекомендация для Group 2/3:**

1. Уменьшить `end` → `+=300%` (сэкономит ~1/4 скролла без потери читаемости шагов).
2. Связать колонки горизонтальными hairline-метками, идущими через обе колонки на уровне активного шага (dotted line + accent-ring) — создаст визуальный мост между шагом и соответствующей точкой на башне.
3. Сбалансировать высоту: либо `min-h` на rail равное высоте scene, либо переход на двухколоночный sticky-layout внутри pinned-зоны, где текст-карточка позиционируется ровно в вертикальном центре напротив активной точки башни.

**Критичность:** M (UX — не блокирует, но воспринимается как «ещё не готово»). **Не фиксим сейчас, готово для Group 2/3.**

### Наблюдение 2 — Hero-карусель: автопауза при клике на картинку не работает

**Код-анализ** (`HeroBgCarousel.tsx`, `useHeroCarousel.ts`):

Это **не регресс, а осознанное изменение** поведения. В коде `HeroBgCarousel.tsx:47–65` явно описано:

> «Pause model — Only one source parks the carousel: hover. Pointer inside → pause, pointer outside → resume. No timers, no grace windows. A click on a number previously added a dedicated pause window (9 s → 5 s in earlier iterations) to let the viewer study their pick, but it felt too static: users saw a non-ticking progress bar and thought the carousel had stalled.»

Текущее поведение:

- **Hover** на зоне продукта (desktop + mobile через `onPointerEnter`) → **пауза**. Progress bar останавливается.
- **Клик на картинку/номер** → goTo(i). Progress bar сбрасывается на 0 и сразу запускается заново — это намеренно, чтобы пользователь видел, что карусель продолжает работать после клика.
- **Play/pause кнопка** (справа, в строке переключателей 01..04) → `toggle()` явная пауза/воспроизведение.

Ранее, судя по комментарию, был «grace window» после клика (9s → 5s). Его явно убрали, потому что пользователи не понимали, застыла карусель или нет.

**Рекомендация:**
- Если заказчик хочет «клик → пауза навсегда до ручного resume» — это отдельный UX-выбор. Нужно решение. Может быть, вернуть `goTo` с `setIsPlaying(false)` на каждый клик, но без автозапуска. Явная кнопка play снова делает isPlaying=true.
- Если заказчик хочет «клик → пауза на N секунд» — нужно определить N. Раньше было 5s, отказались. Может быть, 8–10s?
- Если заказчик доволен hover-pause (и жалуется потому что не заметил play-кнопку) — ничего не делаем, кнопка уже есть справа от номеров.

**Критичность:** M (осознанное решение, но disconnect с ожиданием пользователя). **Требует решения заказчика по целевому поведению перед Group 2/3.**

### Наблюдение 3 — Hero-картинки при resize: текст масштабируется, 3D-рендеры фиксированы

**Код-анализ** (`HeroBgCarousel.tsx`, `ProductHero.tsx`):

Подтверждено. Два места с фиксированным px-размером картинки на mobile-ветке (под `md:hidden`):

- `HeroBgCarousel.tsx:370` — `className="relative h-[200px] w-[200px]"` — **home hero mobile**
- `ProductHero.tsx:217` — `className="relative h-[260px] w-[260px]"` — **product page hero mobile**

На desktop (`>= md`) оба используют проценты (`h-[70%] w-[85%]` для home, `h-[72%] w-[85%]` для product page) от parent — это адекватно респонсив масштабируется при resize desktop-viewport'а.

Проблема — именно на mobile-ветке: 200×200 и 260×260 зафиксированы вне зависимости от ширины экрана (360px / 390px / 414px и т.д.). При устройстве 360×640 картинка занимает 55.5% ширины, при 414×896 — 48.3%. Не катастрофа, но text scales via `clamp()` while image stays pinned.

**Рекомендация для Group 2/3:**

Заменить фиксированные размеры на relative:

```tsx
// Home hero mobile (HeroBgCarousel.tsx:370)
className="relative h-[min(55vw,260px)] w-[min(55vw,260px)]"

// Product page hero mobile (ProductHero.tsx:217)
className="relative h-[min(65vw,300px)] w-[min(65vw,300px)]"
```

Или через `aspect-square` + width в vw с max-width в px.

**Критичность:** L (polish — заметно только при resize DevTools на mobile-размерах, у реальных пользователей фиксированный viewport). **Не фиксим сейчас, готово для Group 2/3.**
